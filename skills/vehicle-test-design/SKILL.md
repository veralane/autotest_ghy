# 车载需求测试设计辅助技能

## 概述

本技能用于辅助AI进行车载系统测试设计分析。基于需求文档，运用专业的测试设计方法，输出结构化的测试设计文档。

**注意：** 本技能为AI辅助分析技能，不提供自动化脚本直接生成测试设计。AI将基于需求内容和测试设计方法，进行分析和推理，输出测试设计建议。

## 输入输出

### 输入：需求文档

需求文档格式参考 `example.xlsx`，包含以下关键字段：

| 字段 | 说明 | 用途 |
|------|------|------|
| 业务编号 | 需求唯一标识 | 测试用例编号追溯 |
| 标题 | 需求标题 | 测试对象名称 |
| 详情 | 需求详细描述 | 测试对象分析依据 |
| 类别 | 需求类别 | 测试类型判断 |
| 需求分类 | 分类标签 | 测试优先级判断 |
| 源ASIL/实现ASIL | 功能安全等级 | 安全测试策略 |
| 安全状态 | 安全状态定义 | 故障测试场景设计 |
| 验证策略 | 验证方法 | 测试方法选择 |
| 验证通过标准 | 通过准则 | 验收标准提取 |
| 验证环境 | 环境要求 | 测试环境设计 |
| 验证条件和限制 | 条件约束 | 测试前置条件 |
| 验证标准 | 标准依据 | 合规性检查 |
| 父需求业务编号 | 上级追溯 | 需求层级分析 |

### 输出：测试设计文档

测试设计文档格式参考 `测试设计模板.md`，包含以下章节：

1. **测试对象分析** - 功能描述、接口、状态、依赖
2. **验收标准** - 功能/性能/安全/合规验收标准
3. **测试点分析** - 正向/异常/边界/安全/接口测试点
4. **DFT需求分析** - 可测试性设计、诊断接口、测试数据需求
5. **测试环境** - 硬件/软件/接口仿真/测试工具
6. **自动化实现分析** - 可行性评估、脚本设计、数据准备

## 测试设计方法

### 一、黑盒测试技术

#### 1. 等价类划分

**原理：** 将输入域划分为若干等价类，每个等价类选取一个代表值进行测试。

**适用场景：**
- 参数取值范围测试
- 输入条件分类测试
- 输出条件分类测试

**分析方法：**
1. 识别输入参数及其约束
2. 划分有效等价类（正常输入）
3. 划分无效等价类（异常输入）
4. 每个等价类选取代表值
5. 生成测试点

**车载参数等价类规则：**

| 参数类型 | 有效等价类 | 无效等价类 |
|----------|------------|------------|
| 数值参数 | 最小值、典型值、最大值 | 最小值-1、最大值+1、null、错误类型 |
| 枚举参数 | 每个枚举值 | 非法枚举值、null |
| 布尔参数 | true、false | null、非布尔值 |
| 数组参数 | 空数组、单元素、典型长度、最大长度 | 超长数组、null |

**示例：**
```
参数: 车速
约束: 0 ≤ 车速 ≤ 200 km/h

有效等价类:
  - EC1: 车速 = 0（静止）
  - EC2: 0 < 车速 < 30（低速）
  - EC3: 30 ≤ 车速 ≤ 120（正常行驶）
  - EC4: 120 < 车速 ≤ 200（高速）

无效等价类:
  - EC5: 车速 < 0（负值）
  - EC6: 车速 > 200（超范围）
  - EC7: 车速 = null（缺省）
```

#### 2. 边界值分析

**原理：** 在等价类边界处选取测试值，边界处更容易出现错误。

**适用场景：**
- 数值范围边界
- 时间边界（超时、周期）
- 状态转换边界
- 容量边界（缓冲区、队列）

**边界值选取策略：**
- **标准边界：** 最小值、最小值+1、典型值、最大值-1、最大值
- **健壮边界：** 最小值-1、最小值、最小值+1、典型值、最大值-1、最大值、最大值+1

**车载常用边界值：**

| 参数类型 | 关键边界值 | 含义 |
|----------|------------|------|
| 车速 | 0, 5, 30, 80, 120, 200 | 静止、ABS退出阈值、ABS激活阈值、典型速度、高速限速、最大车速 |
| 温度 | -40, -20, 0, 25, 40, 85, 125 | 极寒、寒冷、冰点、常温、高温、控制器极限、芯片极限 |
| 电压 | 6, 9, 12, 14.4, 16, 18 | 欠压保护、冷启动、标称、充电电压、过压警告、过压保护 |
| 时间 | 0, 10ms, 100ms, 1s, 10s | 立即响应、实时周期、通信周期、感知阈值、稳定时间 |

**示例：**
```
参数: 制动压力
范围: 0 ~ 100 bar

标准边界测试点:
  - BV1: 0 bar（最小值）
  - BV2: 1 bar（最小值+1）
  - BV3: 50 bar（典型值）
  - BV4: 99 bar（最大值-1）
  - BV5: 100 bar（最大值）

健壮边界测试点:
  - RBV1: -1 bar（无效-下溢）
  - RBV2~RBV6: 同标准边界
  - RBV7: 101 bar（无效-上溢）
```

#### 3. 决策表测试

**原理：** 针对多条件组合的逻辑规则，系统地覆盖所有条件组合。

**适用场景：**
- 多条件判断逻辑
- 状态激活条件
- 故障判定逻辑
- 控制策略切换

**分析方法：**
1. 识别所有条件
2. 识别所有动作
3. 列出所有条件组合
4. 确定每种组合对应的动作
5. 简化决策表（合并等价规则）
6. 生成测试点

**决策表格式：**
```
条件列表: C1, C2, C3, ...
动作列表: A1, A2, A3, ...

| 规则 | C1 | C2 | C3 | 动作 |
|------|----|----|----|------|
| R1   | T  | T  | T  | A1   |
| R2   | T  | T  | F  | A2   |
| ...  | .. | .. | .. | ...  |

注: "-" 表示不关心该条件
```

**示例：**
```
ABS激活条件决策表:

条件:
  C1: 车速 ≥ 30 km/h
  C2: 制动踏板速率 > 阈值
  C3: 无ABS故障
  C4: 轮速传感器正常

动作:
  A1: ABS激活
  A2: ABS不激活
  A3: 进入故障模式

| 规则 | C1 | C2 | C3 | C4 | 动作 |
|------|----|----|----|----|------|
| R1   | T  | T  | T  | T  | A1   |
| R2   | T  | T  | T  | F  | A3   |
| R3   | T  | T  | F  | -  | A3   |
| R4   | T  | F  | T  | T  | A2   |
| R5   | F  | -  | T  | T  | A2   |
```

#### 4. 状态转换测试

**原理：** 基于系统的状态机模型，测试状态转换的正确性。

**适用场景：**
- 系统状态机测试
- 工作模式切换
- 故障状态处理
- 生命周期测试

**分析方法：**
1. 识别所有状态
2. 识别所有转换事件
3. 绘制状态转换图
4. 生成状态转换表
5. 设计覆盖准则
6. 生成测试点

**覆盖准则：**
| 准则 | 定义 | 强度 |
|------|------|------|
| 状态覆盖 | 每个状态至少访问一次 | ★★☆☆☆ |
| 转换覆盖 | 每个转换至少执行一次 | ★★★☆☆ |
| 路径覆盖 | 所有可能路径都执行 | ★★★★★ |

**示例：**
```
TCS系统状态机:

状态:
  S1: 待机状态
  S2: 监控状态
  S3: 激活状态
  S4: 故障状态

转换事件:
  E1: 车辆启动
  E2: 检测到打滑
  E3: 打滑消除
  E4: 检测到故障
  E5: 系统复位
  E6: 车辆熄火

状态转换表:
| 当前状态 | 事件 | 下一状态 | 动作 |
|----------|------|----------|------|
| S1       | E1   | S2       | 进入监控 |
| S2       | E2   | S3       | TCS激活 |
| S2       | E4   | S4       | 记录故障 |
| S3       | E3   | S2       | TCS退出 |
| S3       | E4   | S4       | 记录故障 |
| S4       | E5   | S1       | 故障清除 |
| S2/S3    | E6   | S1       | 系统关闭 |

测试路径:
  TC1: S1→E1→S2→E6→S1（正常启停）
  TC2: S1→E1→S2→E2→S3→E3→S2→E6→S1（正常激活退出）
  TC3: S1→E1→S2→E4→S4→E5→S1（故障处理）
```

#### 5. 场景测试

**原理：** 基于用户实际使用场景，设计端到端的测试场景。

**适用场景：**
- 用户操作流程测试
- 系统集成测试
- 实车道路测试
- 用户验收测试

**场景类型：**
| 类型 | 说明 |
|------|------|
| 基本场景 | 正常用户操作流程 |
| 替代场景 | 替代操作路径 |
| 异常场景 | 错误处理流程 |
| 边界场景 | 极限条件场景 |

**场景描述格式：**
```
场景名称: [场景描述]

前置条件:
  - 条件1
  - 条件2

测试步骤:
  步骤1: [操作] → [预期结果]
  步骤2: [操作] → [预期结果]
  ...

后置条件:
  - 结果状态1
  - 结果状态2
```

**示例：**
```
场景名称: 正常紧急制动

前置条件:
  - 车辆处于正常运行状态
  - ABS系统无故障

测试步骤:
  步骤1: 车辆加速至60km/h → 车速稳定
  步骤2: 快速踩下制动踏板 → ABS激活
  步骤3: 保持制动直至停止 → 车辆平稳停止

预期结果:
  - ABS指示灯亮起
  - 踏板有回弹感
  - 车辆保持直线
  - 制动距离≤40m
```

### 二、功能安全测试（ISO 26262）

#### ASIL等级测试要求

| ASIL等级 | 测试要求 | 覆盖要求 |
|----------|----------|----------|
| ASIL_A | 功能正确性测试、基础故障检测测试 | 分支覆盖 |
| ASIL_B | ASIL_A + 故障检测能力测试、故障响应时间测试 | 分支覆盖 + 故障注入 |
| ASIL_C | ASIL_B + 冗余机制测试、安全状态验证 | MC/DC覆盖 |
| ASIL_D | ASIL_C + 完整安全机制测试、系统级安全验证 | MC/DC覆盖 + 完整故障注入 |

#### 安全测试场景

| 故障类型 | 测试内容 |
|----------|----------|
| 单点故障 | 传感器故障、执行器故障、通信故障、电源故障 |
| 多点故障 | 双传感器同时故障、主备通道同时故障 |
| 潜伏故障 | 未被检测的故障、故障累积场景 |
| 安全状态验证 | 故障后进入安全状态、安全状态维持、故障恢复 |

### 三、故障注入测试

#### 故障类型

| 类型 | 具体故障 |
|------|----------|
| 硬件故障 | 传感器信号丢失/漂移/卡滞、执行器卡滞/失效、电源欠压/过压、通信中断 |
| 软件故障 | 数据越界/损坏、计算溢出、时序异常、内存泄漏 |
| 环境故障 | 温度超限、湿度超限、EMI干扰 |

#### 故障注入方法

| 方法 | 具体操作 |
|------|----------|
| 硬件注入 | 信号断开、信号短路、信号接电阻、信号叠加噪声 |
| 软件注入 | 修改输入数据、修改内部变量、触发异常条件、模拟时序异常 |

#### 故障注入测试用例格式

```
前置条件:
  - 系统正常工作状态
  - 特定工况条件

故障注入:
  - 故障类型: [硬件/软件/环境]
  - 故障描述: [具体故障内容]
  - 注入方式: [注入方法]

预期响应:
  - 检测时间: ≤XXms
  - 故障码: DTC码
  - 系统行为: [降级/禁用/安全状态]
  - 用户提示: [仪表指示]

后置验证:
  - 系统状态验证
  - 功能恢复验证
```

### 四、接口一致性测试

#### 测试内容

| 测试项 | 检查内容 |
|--------|----------|
| 信号一致性 | 名称、类型、范围、单位、分辨率 |
| 时序一致性 | 发送周期、相位关系、超时定义 |
| 协议一致性 | CAN ID、DLC、字节顺序、编码规则 |

#### 测试方法

| 方法 | 说明 |
|------|------|
| 静态检查 | 对比接口文档、检查数据库一致性 |
| 动态测试 | 实际信号交换测试、边界值传输测试、异常值传输测试 |

### 五、时序测试

#### 时序参数

| 参数 | 定义 | 要求 |
|------|------|------|
| 响应时间 | 事件触发到响应输出 | ≤规定时间 |
| 周期精度 | 周期任务执行间隔 | ±抖动范围 |
| 同步精度 | 多信号同步偏差 | ≤同步容差 |
| 超时处理 | 等待超时后的处理 | 进入安全状态 |

### 方法选择指南

#### 根据需求类型选择

| 需求类型 | 首选方法 | 补充方法 |
|----------|----------|----------|
| 功能需求 | 等价类划分 + 边界值分析 | 决策表（复杂逻辑）、状态转换（有状态机） |
| 性能需求 | 边界值分析 | 负载测试、压力测试 |
| 接口需求 | 等价类划分 | 接口一致性测试 |
| 安全需求 | 故障注入测试 | 状态转换、场景测试 |

#### 根据ASIL等级选择

| ASIL等级 | 推荐方法 |
|----------|----------|
| ASIL_A | 等价类 + 边界值 + 语句覆盖 |
| ASIL_B | 等价类 + 边界值 + 决策表 + 分支覆盖 + 故障注入 |
| ASIL_C | 完整黑盒 + MC/DC + 完整故障注入 |
| ASIL_D | 完整黑盒 + MC/DC + 完整故障注入 + 形式化验证 |

## 使用方法

### AI辅助测试设计流程

1. **接收需求**：用户提供需求文档或需求描述
2. **需求解析**：AI解析需求内容，提取关键信息
3. **方法选择**：AI根据需求类型和ASIL等级选择合适的测试设计方法
4. **分析推理**：AI运用测试设计方法进行分析
5. **输出设计**：AI输出结构化的测试设计文档
6. **交互优化**：用户可要求AI补充或调整特定部分

### 示例用法

**场景1：从需求文档进行测试设计**
```
请根据这份ABS功能需求，进行测试设计分析
```

**场景2：针对特定参数进行边界值分析**
```
针对制动压力参数（范围0-100 bar），进行边界值测试点分析
```

**场景3：针对多条件逻辑进行决策表设计**
```
根据ABS激活条件（车速≥30、踏板速率>阈值、无故障、传感器正常），设计决策表测试
```

**场景4：针对状态机进行状态转换测试设计**
```
根据TCS系统状态机，设计状态转换测试
```
